# **【要件定義・技術選定書 v5.0 全文】じゃれ本ランダマイザー**

## **1\. 概要**

### **1.1. プロジェクト名**

じゃれ本ランダマイザー

### **1.2. 目的**

株式会社バーグハンバーグバーグが運営するYouTubeチャンネル「オモコロチャンネル」の人気企画「じゃれ本」を、より新鮮な形で楽しむことを目的とする。  
具体的には、動画内の各作品部分を独立したクリップとして扱い、それらをランダムに再生するウェブアプリケーションを開発する。これにより、視聴済みで内容を記憶しているユーザーに対しても、偶発的で新しい視聴体験を提供する。

### **1.3. ターゲットユーザー**

* オモコロチャンネルの視聴者  
* 特に「じゃれ本」企画のファンで、繰り返し視聴しているユーザー

### **1.4. プロジェクトゴール**

* 「じゃれ本」の作品単位でのランダム再生機能を持つウェブアプリを開発し、公開する。  
* インフラ費用を発生させないため、Cloudflare Pagesの無料プランの範囲内でデプロイし、運用する。

## **2\. 要件定義**

### **2.1. 機能要件**

本アプリケーションが提供すべき機能を以下に定義する。

| ID | 機能名 | 機能概要 | 優先度 | 備考 |
| :---- | :---- | :---- | :---- | :---- |
| F-001 | 作品データ読み込み | 事前に定義された「じゃれ本」の作品リスト（JSON形式）をアプリケーションに読み込む。 | **必須** | データは外部ファイルとして管理する。 |
| F-002 | ランダム作品選択 | 読み込んだ作品リストから、再生する作品をランダムに1つ選択する。 | **必須** | 著者フィルタリング機能（F-011）が有効な場合は、その条件に合致する作品の中から選択する。 |
| F-003 | 動画のタイムスタンプ再生 | 選択された作品のデータに基づき、指定された開始時間(startTime)からYouTube動画の再生を開始する。 | **必須** | YouTube IFrame Player APIを利用。 |
| F-004 | 動画のタイムスタンプ停止 | 再生時間が指定された終了時間(endTime)に達した場合、動画の再生を停止する。 | **必須** |  |
| F-005 | 連続ランダム再生 | 一つの作品の再生が終了（F-004）した後、自動的に次の作品をランダムに選択（F-002）し、再生を開始（F-003）する。 | **必須** |  |
| F-006 | 再生中作品の情報表示 | 現在再生中の作品に関する以下の情報を画面に表示する。 ・元動画のタイトル ・作品タイトル ・作者名 | **必須** | 元動画タイトルは、クリックするとYouTubeの元動画へ遷移するリンクとする。 |
| F-007 | 手動スキップ機能 | ユーザーが任意のタイミングで現在の作品を中断し、次のランダムな作品へ再生を移行させるための「次の作品へ」ボタンを設置する。 | **必須** |  |
| F-008 | 再生/一時停止コントロール | 標準的な動画プレーヤーとして、再生と一時停止を切り替えるボタンを設置する。 | **必須** |  |
| F-009 | 作品タイトルの表示/非表示 | （オプション）作品タイトルを初期状態では非表示にし、ユーザーのアクション（例: クリック）で表示させる機能。 | 推奨 | クイズ的な要素を付加できる。 |
| F-010 | 重複再生の低減 | 連続再生（F-005）の際、直前に再生した作品が再度選択されないようにする。 | 推奨 | より多様な視聴体験を提供するため。 |
| F-011 | 著者フィルタリング機能 | 全著者の一覧をボタン形式で表示する。ユーザーは一人または複数人の著者を選択できる。選択後、ランダム再生の対象が、選択された著者の作品のみに限定される。 | **必須** | 全著者リストは作品データから動的に生成する。「全選択/全解除」機能も提供する。 |
| F-012 | 広告表示機能 | アプリケーション内の指定されたエリアに広告を表示する。この機能はユーザーが任意で有効/無効を切り替えられる。デフォルト（初期状態）は無効（非表示）とする。 | 任意 | 実装にはGoogle AdSense等の広告配信サービスへの登録と審査が別途必要。 |

### **2.2. 非機能要件**

| ID | 種別 | 要件内容 |
| :---- | :---- | :---- |
| NF-001 | パフォーマンス | ・初回アクセス時のページロード時間が3秒以内に完了することを目指す。 ・ボタン操作や作品の切り替えがストレスなく行えること。 |
| NF-002 | ユーザビリティ | ・直感的に操作方法が理解できる、シンプルなインターフェースであること。 ・PC、タブレット、スマートフォンの各デバイスで表示崩れがなく、快適に操作できること（レスポンシブデザイン）。 |
| NF-003 | 可用性 | Cloudflareのインフラを利用し、高い稼働率を維持する。 |
| NF-004 | コスト | Cloudflare Pagesの無料枠に収まる構成とし、サーバー/インフラ費用を一切発生させないこと。 |
| NF-005 | 保守性 | ・コードは適切にコメントが記述され、可読性が高いこと。 ・「じゃれ本」の新作が追加された際、作品データ（JSONファイル）を修正するだけで簡単に追加・更新ができること。 |

### **2.3. 画面・UI要件**

* **レイアウト:**  
  * ページ上部に、設定エリア（著者フィルターなど）を配置する。このエリアはアコーディオンUIなどで開閉可能とすることが望ましい。  
  * 中央にYouTubeプレーヤーを配置する。  
  * プレーヤーの周辺（下部または横）に、現在再生中の作品情報エリアを設ける。  
  * 情報表示エリアのさらに下、または分かりやすい位置にコントロールボタンを配置する。  
  * ページの下部または上部に、広告表示用のエリアを確保する。このエリアは広告表示が無効の場合は非表示とする。  
* **設定エリアのUI:**  
  * **著者フィルター:**  
    * 作品データに存在する全著者名をボタン（チェックボックスのように機能する）として一覧表示する。  
    * 複数の著者ボタンをアクティブ（選択状態）にできる。  
    * 「すべて選択」「すべて解除」の操作を行えるボタンを設置する。  
  * **アプリ設定:**  
    * 「広告を表示する」のON/OFFを切り替えるトグルスイッチを設置する。  
* **デザイン:**  
  * シンプルでクリーンなデザインを基調とする。  
  * テキストは視認性の高いフォントとサイズを選ぶ。

### **2.4. データ要件**

* **データ形式:** JSON (JavaScript Object Notation)  
* **データ管理:** アプリケーション本体とは別の静的ファイル（例: works.json）として管理する。  
* **データ構造（スキーマ）:** 作品リストは、以下のキーを持つオブジェクトの配列とする。  
  \[  
    {  
      "id": 1,  
      "videoId": "YouTube動画のID (例: dQw4w9WgXcQ)",  
      "videoTitle": "元動画の正式タイトル",  
      "workTitle": "作品のタイトル",  
      "author": "作者名",  
      "startTime": 122,  
      "endTime": 255  
    }  
  \]

* **データ作成:** この作品データは、アプリケーション開発とは別に、事前に手動で作成・収集する必要がある。

### **3.2. フロントエンド技術**

* **言語: TypeScript**  
  * **選定理由:** JavaScriptに静的型付けを導入することで、開発中のエラー検出、コード補完の強化、コードの可読性と保守性の向上を図る。RemixはTypeScriptを標準でサポートしており、データの受け渡しを型安全に行えるため、開発体験が大幅に向上する。  
* **フレームワーク: Remix (SPAモード)**  
  * **選定理由:**  
    1. **優れた開発体験:** ファイルベースルーティング、宣言的なデータ取得を担うloader関数など、モダンなWebアプリケーションを構築するための規約が整備されており、コードの見通しが良くなる。  
    2. **Web標準への準拠:** Request/Responseオブジェクトなど、Webの基本技術をベースに設計されており、フレームワークの学習がWeb開発の普遍的な知識習得に繋がる。  
    3. **Cloudflare Pagesとの完全な互換性:** **SPAモード**でビルドすることにより、サーバー不要の静的なファイル群（HTML/CSS/JS）を生成可能。これはCloudflare Pagesの静的ホスティング機能と完璧にマッチし、コストゼロ、高速配信、簡単なデプロイ（NF-001, NF-004）を実現する。  
* **スタイリング: Tailwind CSS**  
  * **選定理由:** ユーティリティファーストのアプローチにより、HTML（JSX）から離れることなく迅速にUIを構築できる。デザインの一貫性を保ちやすく、レスポンシブデザインの実装も容易。  
* **状態管理:**  
  * **選定理由:** コンポーネント単位のUI状態はReact標準のuseStateフックで管理する。ページで必要となるデータ（作品リストなど）はRemixのloader関数で宣言的に取得・管理するため、複雑な状態管理ライブラリは不要。  
* **YouTubeプレーヤー連携: react-youtube**  
  * **選定理由:** YouTube IFrame Player APIをラップした人気のReactコンポーネント。Reactのコンポーネントライフサイクルに沿ってプレーヤーを安全に操作でき、実装を簡素化できる。

### **3.3. API・ライブラリ**

* **YouTube IFrame Player API:**  
  * **選定理由:** JavaScriptから埋め込みYouTubeプレーヤーを精密に制御するための公式API。再生、停止、指定時間へのシークなど、本アプリの根幹となるタイムスタンプ制御機能（F-003, F-004）を実現するために必須。(react-youtubeが内部で使用)

### **3.4. データ管理**

* **静的JSONファイル:**  
  * **選定理由:** データベースを必要とせず、構造が単純で人間にも可読性が高い。Remixのloader関数内からfetch APIを用いて非同期に読み込むことで、コードとデータを綺麗に分離でき、保守性が向上する（NF-005）。

### **3.5. インフラ・デプロイ**

* **バージョン管理: Git / GitHub**  
  * **選定理由:** コードの変更履歴を管理し、Cloudflare Pagesとの連携をスムーズにするため。  
* **ホスティング・デプロイ: Cloudflare Pages**  
  * **選定理由:**  
    * **コスト:** 寛大な無料枠があり、個人開発の静的サイトをホスティングする上で費用が発生しない。  
    * **パフォーマンス:** グローバルなCDNが標準で利用でき、世界中どこからでも高速なアクセスが期待できる。  
    * **CI/CD:** GitHubリポジトリと連携させることで、mainブランチへのプッシュをトリガーに、ビルドとデプロイを自動化できる。  
    * **Remix (SPAモード)との適合性:** 本プロジェクトで生成される静的ファイル群をホスティングするプラットフォームとして最適である。

### **3.6. 外部サービス連携**

* **広告配信サービス (例: Google AdSense)**  
  * **選定理由:** 機能要件F-012（広告表示機能）を実現するため。ウェブサイトのコンテンツに応じた広告を配信し、収益化の可能性を提供する代表的なサービス。  
  * **備考:** 導入には、サービスの利用規約への準拠、サイトの審査、および指定された広告コードの埋め込みが必要となる。

## **4\. 開発・運用計画**

### **4.1. 開発タスクリスト（推奨ステップ）**

1. **【最優先】作品データ作成:** 5〜10件程度の「じゃれ本」作品データを定義済みのJSON形式で作成する。  
2. **環境構築:** npx create-remix@latestコマンドを用いてRemixプロジェクトの雛形を作成し、GitHubリポジトリで管理を開始する。  
3. **基本ルーティングとレイアウト作成:**  
   * Remixの規約に沿ってルートファイルを作成する。  
   * react-youtubeを配置し、固定の動画が再生できることを確認する。  
   * Tailwind CSSを使い、プレーヤー、情報表示エリア、ボタンの基本的な配置を実装する。  
4. **コアロジック実装:**  
   * Remixのloader関数を用いてpublicディレクトリ内のworks.jsonファイルを読み込む機能を実装する。  
   * loaderから渡されたデータに基づき、作品のランダム選択と、情報表示エリアへの反映機能を実装する。  
   * 選択された作品のstartTimeから再生を開始し、endTimeを監視して再生を停止するロジックを実装する。  
5. **コントロール機能実装:**  
   * 「再生/一時停止」「次の作品へ」ボタンに、対応する機能を実装する。  
   * 連続再生機能を実装する。  
6. **デプロイと動作確認:** ここまでを基本バージョンとし、Cloudflare Pagesにデプロイして動作を確認する。  
7. **追加機能開発フェーズ:**  
   * **著者フィルター機能:** loaderで全著者のリストを生成し、UIを構築。選択された著者に応じてloaderが返す作品リストを絞り込むロジックを実装する。  
   * **広告機能実装:** 広告設定スイッチの状態に応じて、広告コードをマウントするコンポーネントの表示/非表示を制御する。  
8. **仕上げ:**  
   * 全機能を含めたレスポンシブ対応と最終的なデザイン調整を行う。  
   * （任意）オプション機能（F-009, F-010）を実装する。  
9. **最終デプロイ:** 全機能を実装したバージョンをデプロイし、最終確認を行う。

### **4.2. 運用・保守**

* **作品データの更新:** 「じゃれ本」の新作が公開された場合、works.jsonファイルに新しい作品データを追記し、GitHubリポジトリにプッシュする。Cloudflare Pagesが自動で再デプロイを行い、更新が反映される。  
* **不具合修正:** 不具合が発見された場合は、コードを修正し、同様にGitHub経由でデプロイする。
